## Why It Matters

`unity-mcp` is a mature example of an MCP server that does not merely wrap HTTP APIs. It bridges an AI coding agent into a stateful GUI/game editor, where correctness depends on live editor state, main-thread execution, compilation, domain reloads, asset imports, and visual verification. That makes it highly relevant for Agentic Coding Lab patterns around domain-specific coding support, context control, verification, and safe mutation.

It also shows what happens when a domain bridge grows beyond a few tools: it needs tool grouping, resource-first guidance, routing to the right runtime instance, batching, async job polling, and client-specific configuration. Those are reusable patterns for any coding agent that must operate inside a complex local toolchain rather than only editing files.

## What It Is

MCP for Unity is a Unity Editor package plus a Python FastMCP server. The Unity side owns editor state and executes commands through C# tool/resource handlers. The Python side exposes MCP tools/resources, handles client transports, routes calls to the selected Unity instance, normalizes responses, and writes MCP client configuration for tools like Cursor, Claude Code, Codex, VS Code, Windsurf, Cline, OpenCode, Gemini CLI, Qwen Code, and others.

The repo contains:

- `MCPForUnity/`: Unity package with editor windows, transports, tool/resource discovery, tool implementations, client configurators, security settings, and helper services.
- `Server/`: Python package `mcpforunityserver`, FastMCP entrypoint, tool/resource wrappers, HTTP/WebSocket bridge, legacy stdio/TCP bridge, API-key auth, CLI, and tests.
- `unity-mcp-skill/`: bundled coding-agent skill with resource-first workflows, tool reference, and usage recipes.
- `TestProjects/UnityMCPTests/`: Unity test project covering C# editor tools, transport services, helpers, resources, and play/edit mode behaviors.
- `.github/workflows/`: Python test, Unity test, release, and live natural-language/Unity suite workflows.

## Research Themes

- Token efficiency: Good. It uses resource-first instructions, `batch_execute`, paginated hierarchy/component/file reads, dynamic tool groups, and a bundled skill that tells agents to inspect before mutating. The server also hides non-default HTTP tool groups until Unity registers enabled tools or the agent calls `manage_tools`.
- Context control: Strong. It separates read-only MCP resources from mutating tools, injects `unity_instance` into request context, supports per-session active instance selection, exposes editor readiness via `mcpforunity://editor/state`, and uses tool groups to keep irrelevant capabilities out of tool listings.
- Sub-agent / multi-agent: Not a subagent system. The closest patterns are multi-Unity-instance routing and remote-hosted per-user session isolation through `(user_id, project_hash)` registry keys.
- Domain-specific workflow: Very strong. The bridge understands Unity-specific constraints: main-thread execution, asset database refresh, compilation/domain reload, prefab stages, packages, render pipelines, tests, screenshots, ProBuilder, physics, graphics, profiler, and live API reflection.
- Error prevention: Strong but uneven. It has preflight checks, retry hints, transport timeouts, status files, SHA preconditions for script edits, path traversal/symlink guards under `Assets/`, Roslyn/basic validation, package dependency checks, and remote API-key auth. It still exposes broad mutation once a client is connected.
- Self-learning / memory: Limited. It stores session/tool/job/editor preferences and telemetry UUID/milestones, but no agent learning or durable semantic memory.
- Popular skills: Bundled `unity-mcp-skill` is useful. It gives agents operational guidance: read `editor_state`, batch repetitive operations, wait for compilation, check console, use screenshots, paginate large queries, and verify Unity APIs with reflection/docs.

## Core Execution Path

HTTP transport path:

1. MCP client calls the FastMCP server at `/mcp`, usually from config generated by the Unity editor package.
2. Unity Editor connects separately to `/hub/plugin` via WebSocket.
3. `PluginHub.on_connect` accepts the Unity plugin, optionally validating `X-API-Key` in remote-hosted mode.
4. Unity sends `register` with project name/hash/version/path, then `register_tools` with enabled tool metadata.
5. The server stores sessions in `PluginRegistry`, syncs FastMCP tool visibility, and sends `tools/list_changed` to active MCP clients.
6. `UnityInstanceMiddleware` injects `user_id`, active `unity_instance`, and `unity_session_id` into tool/resource calls.
7. Python tool wrappers call `send_with_unity_instance`, which routes HTTP-mode calls through `PluginHub.send_command_for_instance`.
8. `PluginHub` sends an `execute` message over WebSocket to Unity.
9. `WebSocketTransportClient` receives it, wraps it as `{type, params}`, and calls `TransportCommandDispatcher.ExecuteCommandJsonAsync`.
10. `TransportCommandDispatcher` runs the command on Unity's main thread, blocks disabled tools/resources, and calls `CommandRegistry.ExecuteCommand`.
11. `CommandRegistry` invokes the C# handler discovered via `[McpForUnityTool]` or `[McpForUnityResource]`, then returns JSON through WebSocket to the MCP client.

Stdio transport path:

1. MCP client launches `uvx --from mcpforunityserver mcp-for-unity --transport stdio`.
2. Unity's `StdioBridgeHost` auto-starts a loopback TCP listener, writes `~/.unity-mcp/unity-mcp-status-<hash>.json`, and sends strict `FRAMING=1` handshakes.
3. Python `UnityConnectionPool` scans status/port files, probes loopback listeners, connects by project name/hash/port, and sends framed JSON commands.
4. Unity queues framed commands and executes them through the same `TransportCommandDispatcher` and `CommandRegistry`.
5. Python retries across domain reloads, rediscovering ports and returning structured `hint="retry"` responses when Unity is reloading or busy.

Long-running tool path:

- C# tools can return `PendingResponse` with `_mcp_status` and polling metadata.
- Python `CustomToolService` and tool wrappers poll the configured `PollAction` until completion/error/timeout.
- Unity persists some job state through `McpJobStateStore`, `PackageJobManager`, and `TestJobManager`, so domain reloads do not always lose progress.

## Architecture

The design is a two-process bridge:

- Python MCP facade: `Server/src/main.py` creates the FastMCP server, registers tools/resources, mounts `/hub/plugin`, manages HTTP/stdin transport, exposes local REST helper routes, and configures remote API-key auth.
- Python registries: `services/registry/tool_registry.py` and `resource_registry.py` auto-register Python wrappers. `PluginRegistry` tracks live Unity sessions. `CustomToolService` tracks project/global custom tools.
- Python middleware: `UnityInstanceMiddleware` controls instance selection, remote user isolation, inline `unity_instance` routing, and HTTP-mode tool-list filtering.
- Unity transport: `TransportManager` abstracts HTTP WebSocket and stdio bridge clients. `WebSocketTransportClient` handles registration, keepalive, reconnect, and tool re-registration. `StdioBridgeHost` handles the legacy framed TCP listener and status files.
- Unity execution: `TransportCommandDispatcher` guarantees main-thread execution. `CommandRegistry` reflects loaded assemblies for `[McpForUnityTool]` and `[McpForUnityResource]` classes and dispatches `HandleCommand(JObject)`.
- Unity capabilities: tool handlers cover scene, GameObject, component, script, asset, material, package, build, test, camera, graphics, physics, animation, UI, VFX, ProBuilder, profiler, docs, and editor operations.
- Client configuration: Unity writes JSON/TOML/CLI config blocks for many MCP clients, preserving existing files where possible and using atomic writes.
- Verification: Python pytest covers server tools, transport/auth, registries, parameter normalization, telemetry, and CLI. Unity edit-mode tests cover C# services/tools/resources. GitHub workflows run Python tests and Unity tests; a manual live NL/T suite starts Unity in Docker and drives the MCP bridge with natural-language tasks.

## Design Choices

Python is mostly a schema/router layer; Unity performs actual editor mutations. This keeps domain logic near Unity APIs and avoids reimplementing Unity semantics in Python.

Tool and resource discovery is attribute/decorator driven on both sides. Python modules use `@mcp_for_unity_tool` and `@mcp_for_unity_resource`; Unity classes use `[McpForUnityTool]` and `[McpForUnityResource]`. This makes extension easy and lets Unity report dynamic custom tools back to the server.

Resources are treated as the inspection layer and tools as the mutation layer. The server instructions and bundled skill explicitly tell agents to read resources before taking action.

Tool visibility is context control, not just UI polish. Groups like `core`, `docs`, `vfx`, `animation`, `ui`, `testing`, `probuilder`, and `profiling` can be enabled/disabled per session, mirrored from Unity editor toggles, and pushed through `tools/list_changed`.

Multi-instance routing is first-class. Agents can read `mcpforunity://instances`, call `set_active_instance`, or pass `unity_instance` inline. Remote-hosted mode scopes sessions by user and disables unsafe auto-selection.

Mutation workflows include evidence and recovery loops. `editor_state` returns readiness/advice/staleness, script edits require SHA preconditions, preflight waits for compilation where needed, `read_console` closes the loop after script changes, and screenshot tools provide visual verification.

The repo accepts high capability in exchange for local power. Once trusted and connected, the agent can modify scenes, assets, scripts, packages, builds, and even run C# code if scripting extensions are enabled.

## Strengths

- Excellent real-world domain bridge: handles live editor state, GUI/editor constraints, domain reload, compilation, asset import, test jobs, and visual feedback.
- Strong context-control pattern: resources, tool groups, dynamic visibility sync, and skill guidance reduce tool overload.
- Good multi-instance and remote-user model: project hashes, explicit instance selection, and `(user_id, project_hash)` isolation are worth copying.
- Robust transport work: strict framed stdio/TCP bridge, WebSocket keepalive/reconnect, stale-session eviction, pending-command cancellation, and loopback fallback handling.
- Practical coding-agent workflow support: script edit preconditions, compile/read-console loop, Unity API reflection/docs, screenshots, and batched operations.
- Broad verification surface: Python tests, Unity edit/play mode tests, transport/auth tests, GitHub workflows, and live NL/T workflow.
- Extension story is useful: project teams can add custom C# editor tools with attributes and expose them through MCP without hand-writing Python schemas.

## Weaknesses

- Permission model is coarse. Tool toggles and remote API keys control visibility/access, but there is no per-call approval layer for destructive actions like deleting assets, installing packages, editing scenes, changing build settings, or executing code.
- `execute_code` is intentionally powerful. It compiles and invokes arbitrary C# inside Unity with only pattern-based safety checks, and callers can disable those checks. This should stay opt-in and should not be exposed in shared or untrusted contexts.
- Remote-hosted hardening has one notable gap in reviewed source: `CustomToolService` registers a `/register-tools` HTTP route without an explicit API-key check, while remote auth docs only list `/api/command`, `/api/instances`, and `/api/custom-tools` as disabled. Even if this mostly registers metadata, a public deployment should protect or disable that route.
- Client config automation writes into user-level dotfiles and client configs. It uses atomic writes and tries to preserve existing config, but the behavior is still high-impact and should be explicit in stricter agent environments.
- Server internals rely on FastMCP private attributes/methods for WebSocket route mounting, session tracking, and tool-visibility transforms. Useful, but brittle across FastMCP releases.
- Schema duplication exists between Python wrappers and Unity C# handlers. Drift risk is mitigated by tests and runtime normalization, but still raises maintenance cost.
- `batch_execute` improves latency but is not transactional; earlier Unity mutations are not rolled back if a later command fails.
- At reviewed commit, version signals are slightly skewed: `MCPForUnity/package.json` reports `9.6.9-beta.7`, while `Server/pyproject.toml` and root MCP bundle manifest report `9.6.8`.

## Ideas To Steal

- Build domain MCP bridges as "thin MCP facade + native plugin executor" instead of putting all domain logic in the server.
- Make editor/runtime readiness a canonical resource, including `ready_for_tools`, blocking reasons, staleness, and retry advice.
- Use dynamic tool groups as context control. Keep default tools small, let agents activate specialist groups, and push tool-list changes to clients.
- Support explicit runtime instance routing through a resource, a session-level setter, and per-call override.
- Provide a batch tool for repetitive operations, but keep execution deterministic on the domain main thread.
- Require SHA or equivalent preconditions for code edits that target live project files.
- Pair every script/code workflow with compile polling and console verification.
- Ship an agent skill alongside the MCP server so agents learn domain workflows instead of raw tool names only.
- Model remote shared servers around authenticated user-scoped runtime sessions, not just one global runtime pool.
- Use live reflection/docs tools where domain APIs are version-sensitive and LLM prior knowledge is often stale.

## Do Not Copy

- Do not expose arbitrary in-editor code execution by default. If needed, put it behind explicit local-only or admin-only policy with audit logs.
- Do not leave legacy or helper HTTP routes unauthenticated in remote-hosted deployments.
- Do not depend on private framework internals unless you pin versions and add compatibility tests.
- Do not rely on regex/pattern safety checks as a security boundary for code execution or destructive file operations.
- Do not auto-edit user config files silently. Require clear user intent and preserve backups.
- Do not treat workflow examples in skills as authoritative code; the repo itself warns they are templates that must be verified against project state.
- Do not assume batching is safe for dependent operations unless failures and partial state are explicitly handled.

## Fit For Agentic Coding Lab

This is one of the better repo examples for "agentic coding inside a specialized IDE/editor." It is directly useful as a pattern source for:

- Domain-specific MCP adapters.
- Resource-first context retrieval.
- Runtime/editor readiness checks.
- Tool visibility and token budgeting.
- Instance routing.
- Live verification loops.
- Agent skills attached to a tool server.

It should not be adopted wholesale as a security model. Agentic Coding Lab should borrow the execution architecture and context-control patterns, then add stricter per-tool permissions, explicit approvals for destructive operations, and remote-route hardening.
