## Why It Matters

This repo is a useful contrast to general agent instruction packs because it packages domain expertise as installable skills for web, frontend, mobile, deployment, and Vercel operations. The highest-value example is `vercel-optimize`: it is not just prose. It defines a full agent workflow with observable signals, deterministic candidate selection, scoped investigation briefs, sub-agent output contracts, sanitizers, verification, and final report rendering.

For Agentic Coding Lab, the repo shows how domain-specific skills can move from "remember these best practices" to "run this evidence pipeline, then let the agent investigate only bounded targets." It also shows a lighter-weight pattern for guideline skills: source rules stay in small Markdown files, `SKILL.md` stays compact, generated `AGENTS.md` carries the full expanded rule corpus, and zip artifacts make the skill portable.

## What It Is

`vercel-labs/agent-skills` is Vercel's public collection of agent skills following the Agent Skills / skills.sh layout. Top-level `README.md` catalogs skills, `AGENTS.md` defines how contributors add new skills, and `skills/<skill-name>/` contains the runtime artifacts.

Reviewed skill families:

- `vercel-optimize`: observability-first Vercel cost/performance audit with Node scripts, libraries, references, playbooks, gates, scanners, verification, and a separate test package.
- `react-best-practices`: React and Next.js performance rule pack generated from individual rule files.
- `react-native-skills`: React Native and Expo best-practice rule pack generated by the same build tooling.
- `composition-patterns`: React API/composition rule pack generated by the same build tooling.
- `react-view-transitions`: focused React/Next.js View Transition API guide with reference files and compiled `AGENTS.md`.
- `web-design-guidelines`: thin skill that fetches Vercel's latest Web Interface Guidelines at use time.
- `deploy-to-vercel` and `vercel-cli-with-tokens`: deployment workflows, CLI state checks, token handling rules, and no-auth claimable deploy scripts.

## Research Themes

- Token efficiency: Strong progressive disclosure. Skill descriptions are loaded first, `SKILL.md` points to references only when needed, and heavy details live in `references/`, `rules/`, generated `AGENTS.md`, or executable scripts. `vercel-optimize` pushes large data into JSON artifacts rather than chat context.
- Context control: Very strong in `vercel-optimize`. It forbids source reading before metrics, gates candidates mechanically, resolves a bounded file list for each candidate, and treats repo-wide grep as a malformed-candidate signal. Other skills rely more on human/agent discipline.
- Sub-agent / multi-agent: `vercel-optimize` has a reusable fan-out contract: manifest, one brief per candidate, candidate refs, raw output directory, strict collector, and ordered recommendation array. It hides sub-agent terminology from customer output.
- Domain-specific workflow: Excellent for Vercel and frontend domains. It encodes concrete platform signals, CLI commands, framework version rules, cache semantics, deployment paths, and React/RN implementation patterns.
- Error prevention: Best-in-class parts are deterministic gates, candidate schema validation, docs allow-list, citation/version sanitizers, claim verification, quality floor, public release safety tests, redaction tests, and report rendering caps.
- Self-learning / memory: No self-learning loop or durable memory. The repo favors curated static references, fixtures, and tests over adaptive memory.
- Popular skills: `vercel-optimize`, `react-best-practices`, `web-design-guidelines`, `react-native-skills`, `react-view-transitions`, `composition-patterns`, `deploy-to-vercel`, and `vercel-cli-with-tokens`.

## Core Execution Path

The general skill packaging path is simple: a skill directory contains `SKILL.md`, optional `metadata.json`, optional `AGENTS.md`, optional `rules/`, optional `references/`, optional `scripts/`, and optional `lib/`. Top-level `AGENTS.md` tells contributors to keep `SKILL.md` under 500 lines, use specific trigger descriptions, prefer scripts over inline code, write JSON to stdout and status to stderr, and document skills.sh installation.

For rule-pack skills, the source-of-truth path is:

1. Authors edit `skills/<skill>/rules/*.md` and `_sections.md`.
2. `packages/react-best-practices-build/src/parser.ts` parses rule title, impact, examples, references, and section from Markdown/frontmatter.
3. `src/build.ts` groups rules, assigns stable IDs, reads metadata, and writes compiled `skills/<skill>/AGENTS.md`.
4. `src/validate.ts` checks required title, explanation, code examples, and impact enum.
5. `src/extract-tests.ts` extracts bad/good examples into `test-cases.json` for evaluation-style reuse.

For `vercel-optimize`, the execution path is much richer:

1. `collect-signals.mjs` checks Vercel CLI/auth/project linkage, detects framework support, probes Observability Plus, collects project config, contract, usage, stack, and a shared 14-day set of Vercel metrics.
2. `scan-codebase.mjs` enumerates route files for Next.js, SvelteKit, Nuxt, Astro, and related layouts, runs scanners, resolves workspace imports in monorepos, and emits route/finding JSON.
3. `merge-signals.mjs` combines metrics and scan output, annotating scanner findings with observed route traffic, `COLD-PATH`, or `NO-ROUTE-MAPPING`.
4. `gate-investigations.mjs` runs pure-JS gates, validates candidate contracts, applies hard gates, dedupes segment-tree route aliases, and budgets code-scope candidates.
5. `deep-dive.mjs` enriches launched candidates with extra metric slices, deduping CLI queries and extracting from broad-pass metrics where possible.
6. `reconcile-candidates.mjs` turns disproven candidates into observations before any source investigation.
7. `prepare-investigation-brief.mjs` creates a manifest and one candidate-bound brief with files, citations, support topics, playbooks, evidence, and output schema.
8. Agents investigate the brief and emit one JSON recommendation or one JSON no-change finding.
9. `collect-sub-agent-outputs.mjs` extracts JSON from raw outputs, checks candidateRef coverage, rejects missing/duplicate/unknown outputs, and preserves manifest order.
10. `verify-and-regen.mjs` extracts claims, verifies files/snippets/citations/project facts, applies sanitizers, grades quality, and emits regen plans for weak or unsafe recommendations.
11. `render-report.mjs` dedupes ready recommendations, holds back unsafe ones, renders customer Markdown, and emits a final-message JSON body.

For `deploy-to-vercel`, the path starts with four state checks: git remote, `.vercel/project.json` or `.vercel/repo.json`, CLI auth, and teams. The skill chooses git-push deploy for linked git projects, direct `vercel deploy` for linked no-remote projects, `vercel link` plus deploy for authenticated unlinked projects, and claimable tarball upload scripts as a sandbox fallback.

## Architecture

The repo is a skill bundle, not a conventional app. There is no root package manifest; each complex subsystem carries its own package or scripts.

Important architectural pieces:

- `skills/`: installable skill directories and zipped artifacts.
- `skills/vercel-optimize/SKILL.md`: compact orchestrator instructions with the full Vercel audit pipeline.
- `skills/vercel-optimize/scripts/`: CLI entrypoints for collection, scanning, gating, brief generation, verification, and rendering.
- `skills/vercel-optimize/lib/`: reusable gates, scanners, Vercel CLI wrappers, citation handling, route normalization, report rendering, claim extraction, sanitizers, and quality scoring.
- `skills/vercel-optimize/references/`: doctrine, data collection, recommendation schema, verification rules, support topics, playbooks, docs allow-list, and generated gate/scanner docs.
- `packages/vercel-optimize-tests/`: Node test suite and sanitized fixtures kept outside the installable skill to reduce shipped size.
- `packages/react-best-practices-build/`: TypeScript build tooling shared by the React, React Native, and composition rule packs.
- `.github/workflows/react-best-practices-ci.yml`: CI validates only the React best-practices build package on relevant changes.

Artifact formats are mostly plain Markdown and JSON. The executable pipeline passes JSON artifacts such as `vercel-signals.json`, `codebase.json`, `signals.json`, `gate.json`, `investigation-evidence.json`, `briefs-manifest.json`, `recommendations.json`, `verify.json`, `report.md`, and `final-message.json`. Portable skill artifacts also exist as `.zip` files for several skills. Zip contents mirror skill directories for rule/guideline/deploy skills, but `vercel-optimize` is present as an unpacked directory only in the reviewed tree.

## Design Choices

The most important design choice is evidence-first domain narrowing. `vercel-optimize` does not let the agent begin with source browsing. Metrics and deterministic gates create the investigation target first, then source files are allowed only if they are named by a candidate or route-local import chain.

The second major choice is separating agent judgment from mechanical contracts. Candidate selection is pure JavaScript; sub-agents receive bounded briefs; outputs are JSON records; collection validates candidate refs; verification checks claims mechanically; rendering filters stale or unsafe results. This turns an LLM-heavy workflow into a pipeline with typed handoffs.

The third choice is version-aware domain grounding. `references/docs-library.json` stores allowed URLs, applicability by framework/version, and topic/kind mapping. Citation sanitizers strip unknown or version-mismatched URLs, and tests explicitly probe fabricated URLs, Next 15-only APIs against Next 13/14, and skill-rule citation formats.

The fourth choice is keeping installable skills small. `vercel-optimize` keeps tests outside the skill. Rule-pack skills keep individual rules as source and generated `AGENTS.md` as the expanded agent-facing artifact. `web-design-guidelines` goes further by fetching latest external guidelines at runtime.

The fifth choice is customer-output hygiene. Internal words like "sub-agent", "gate", "passRate", "quality score", and "LLM" are banned from final Vercel Optimize customer copy. Reports surface ready recommendations, observations, investigated/no-change items, not-investigated signals, strengths, and data gaps instead of raw internal state.

## Strengths

- Strong reusable skill contract: `SKILL.md` for activation, `references/` for progressive disclosure, `scripts/` for deterministic work, stdout JSON/stderr status, and Markdown/JSON artifacts that can be inspected independently.
- Excellent context discipline in `vercel-optimize`: metrics first, deterministic gates, candidate-bound file scopes, generated briefs, and explicit refusal to widen scope when a candidate is malformed.
- Practical domain modeling: Vercel metrics, framework support, route mapping, billing dimensions, cache semantics, Observability Plus blockers, project config facts, and deployment linkage are all first-class.
- Robust hallucination controls: allow-listed citations, framework semver matching, sanitizer trail, exact money-literal stripping, claim verification, quality floor, hard safety regen triggers, and report-time stale-candidate filtering.
- Good multi-agent handoff format: manifest, candidateRef, ordered output collection, duplicate/missing checks, and JSON recommendation/no-change schema.
- Useful build workflow for frontend rule packs: individual Markdown rules compile into full `AGENTS.md`, validation enforces shape, and bad/good examples can become evaluation cases.
- Tests cover many failure modes with sanitized real CLI fixtures: gates, scanners, route normalization, observability blockers, redaction, docs freshness, citations, sanitizer behavior, report rendering, and verification/regeneration.

## Weaknesses

- CI coverage is uneven. The only reviewed GitHub Actions workflow validates the React best-practices build package. `vercel-optimize` has many tests and docs checks, but the reviewed workflow does not run them on PRs.
- Catalog metadata drifts. Top-level README and skill metadata disagree on some rule counts and names, such as React "40+" versus `SKILL.md` "70 rules", and React Native catalog naming versus `react-native-skills`.
- Zip artifacts create drift risk. Several skills ship checked-in `.zip` files with historical timestamps and no reviewed manifest/checksum contract tying them to current directories.
- `web-design-guidelines` depends on a remote raw GitHub URL at runtime. That keeps rules fresh, but it weakens reproducibility and offline use unless callers snapshot the fetched guideline version.
- Deployment fallback scripts parse JSON with `grep`/`cut` and poll the deployed preview URL, while the skill prose later says not to fetch deployed URLs when reporting. This is workable for a fallback, but less rigorous than the JSON handling in `vercel-optimize`.
- `vercel-cli-with-tokens` correctly forbids `--token`, but discovery commands like `printenv VERCEL_TOKEN` and `.env` grep can expose secrets in tool output or logs if the host is not careful.
- Some generated or expanded documents are very large. Progressive disclosure mostly mitigates this, but agents that load compiled `AGENTS.md` wholesale can still burn context quickly.

## Ideas To Steal

- Adopt the `vercel-optimize` pipeline shape for any high-stakes domain skill: collect signals, merge local scan, gate candidates, create bounded briefs, collect JSON outputs, verify claims, then render.
- Define a candidate contract for every investigation system: `kind`, `scope`, `route` or `file` target, `files`, `priority`, `confidence`, `reason`, `question`, and stable `candidateRef`.
- Keep "why investigate this?" out of the LLM. Encode threshold expressions in testable code and surface skipped candidates with reasons.
- Use docs allow-lists with framework/version applicability. Pair them with citation sanitizers and tests for fabricated/stale URLs.
- Treat scanner findings as supplementary unless they are traffic-independent or mapped to observed hot paths.
- Split rule packs into small source rules and generated compiled docs. Extract bad/good code examples as evaluation seeds.
- Put tests and fixtures outside installable skills when install size matters, but keep the test paths obvious and documented.
- Generate sub-agent briefs with only the allowed files, filtered citations, relevant playbooks, support topics, and required JSON schema.
- Render customer output from verified artifacts, not from raw sub-agent summaries.
- Hide internal pipeline terms in final output; expose actionable categories and data gaps instead.

## Do Not Copy

- Do not rely on checked-in zip files without a manifest, checksum, or build freshness test.
- Do not leave major skill test suites outside CI if the skill is intended to be production-grade.
- Do not let catalog docs drift from generated skill metadata; add a docs consistency test if the README is a user-facing registry.
- Do not copy remote-fetch-only guideline skills for domains where reproducibility or offline operation matters.
- Do not use shell `grep`/`cut` JSON parsing for security-sensitive or production-critical deploy paths when a structured parser is available.
- Do not ask agents to print secrets while "locating" tokens; prefer presence checks and redacted confirmation.
- Do not copy the full Vercel Optimize complexity for simple guideline packs. Its pipeline is justified by billing/performance risk, not by every domain.

## Fit For Agentic Coding Lab

Fit is high. This repo should be mined for reusable artifact contracts and verification patterns rather than copied wholesale.

Best direct fits:

- A generic `candidate-gate-brief-verify-render` template for domain-specific coding audits.
- A `docs-library.json` schema with `appliesTo` and version constraints for every skill that recommends framework APIs.
- A rule-pack build system that compiles small Markdown rules into agent-facing docs and test/eval examples.
- A sub-agent output collector that accepts raw wrapped JSON but enforces coverage, uniqueness, and stable ordering.
- A "not investigated" report section sourced from deterministic gates to build trust without over-investigating.

Less direct fits:

- Vercel-specific metric IDs, product gates, and billing language are too domain-bound to reuse directly.
- Claim verifier details are valuable as examples, but each target domain will need its own claim extractors and safety checks.

## Excluded Paths

- `.git/` internals and remote feature branches: not needed beyond recording `origin/main` commit.
- Binary zip payload bodies and nested `Archive.zip`: inspected file listings and sizes only; did not unpack every archived byte because directories contain the readable source.
- Full generated `AGENTS.md` bodies for large rule packs: sampled through source rules, build tooling, metadata, and `SKILL.md` to avoid reproducing long generated prompt bodies.
- Most sanitized fixture JSON/TXT contents under `packages/vercel-optimize-tests/test/fixtures/real-cli-output/`: inventoried and sampled through tests/docs; exhaustive fixture review was not necessary for skill contract assessment.
- Individual support-topic and playbook bodies not tied to packaging or verification contracts: sampled representative docs only.
- Unrelated remote branches and unmerged work: review target was current `origin/main` at the recorded commit.
