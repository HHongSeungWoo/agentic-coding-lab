## Why It Matters

OpenSpec is one of the largest public spec-driven development instruction systems for AI coding agents. It is not just a prompt pack: it ships a TypeScript CLI that installs skills and slash commands for many tools, creates durable change folders, emits schema-aware instructions, validates specs, and archives changes into source-of-truth specs.

For Agentic Coding Lab, it is useful because it shows how to make an agent workflow portable across coding assistants while keeping execution state in plain files. The strongest pattern is using the CLI as a narrow state and instruction oracle, while leaving creative artifact creation and implementation to the agent.

## What It Is

OpenSpec is an npm-distributed CLI (`@fission-ai/openspec`) for AI-native spec-driven development. `openspec init` creates an `openspec/` directory and generated skill/command files for selected AI tools. Users then drive work through `/opsx:*` commands such as `propose`, `apply`, `sync`, and `archive`.

The default workflow creates:

- `proposal.md` for why and scope.
- `specs/auth/spec.md`-style capability folders for delta requirements.
- `design.md` for implementation approach.
- `tasks.md` for checkbox-tracked implementation.

The repo also includes a schema system so teams can define different artifact graphs, templates, and apply rules.

## Research Themes

- Token efficiency: Uses profiles (`core` vs `custom`) and delivery modes (`skills`, `commands`, `both`) to avoid installing every workflow by default. Project context has a hard 50KB cap and per-artifact rules prevent every instruction from carrying every rule. However, injected context is still coarse-grained and not ranked, summarized, or budgeted per task.
- Context control: Strong. `openspec instructions proposal --json`-style calls separate `context`, `rules`, `template`, dependencies, output path, and unlocks. Agents are told to read dependency artifacts and not copy context/rules into outputs. Workspace support also separates portable shared state from machine-local paths.
- Sub-agent / multi-agent: Limited but notable. Bulk archive instructions ask the agent to resolve conflicting parallel changes by inspecting implementation evidence. Archive instructions mention using a subagent to run sync, but this is prompt-level orchestration, not a runtime scheduler. Workspaces support cross-repo planning visibility.
- Domain-specific workflow: Strong. The built-in `spec-driven` schema models proposal, specs, design, tasks, and apply state. Custom schemas let teams define their own artifact types and dependencies.
- Error prevention: Strong on filesystem, schema, and spec merge boundaries. Validation catches malformed delta specs, missing scenarios, duplicate requirements, bad dependencies, cycles, path traversal in change names, and invalid workspace state. Archive prepares all spec updates and validates rebuilt specs before writing.
- Self-learning / memory: Uses durable specs and archived changes as project memory. Main `openspec/specs/` is source of truth; `openspec/changes/archive/` preserves audit history. No model-personal memory, retrieval ranking, or automatic lessons learned loop.
- Popular skills: Default core profile installs `openspec-propose`, `openspec-explore`, `openspec-apply-change`, `openspec-sync-specs`, and `openspec-archive-change`. Expanded workflow adds `new`, `continue`, `ff`, `verify`, `bulk-archive`, and `onboard`.

## Core Execution Path

1. Package entrypoint `bin/openspec.js` imports `dist/cli/index.js`; source CLI is `src/cli/index.ts`.
2. `openspec init` validates target path, detects existing tool directories, optionally cleans legacy OpenSpec artifacts, creates `openspec/specs`, `openspec/changes`, and `openspec/changes/archive`, then generates skills and commands based on global profile and delivery mode.
3. Generated skills and commands come from `src/core/templates/workflows/*.ts`; `src/core/shared/skill-generation.ts` wraps skill templates in frontmatter and maps workflow IDs to skill directories.
4. Tool-specific command files are generated by `src/core/command-generation/*` adapters. The adapter layer handles file path and frontmatter differences, while command body content stays tool-agnostic.
5. `openspec new change add-dark-mode` creates `openspec/changes/add-dark-mode/` and writes `.openspec.yaml` with selected schema and created date.
6. `openspec status --change add-dark-mode --json` resolves the schema, scans artifact output paths, and returns `done`, `ready`, or `blocked` artifact state.
7. `openspec instructions proposal --change add-dark-mode --json` returns structured instructions with template, schema instruction, config context, artifact-specific rules, dependency files, and output path. The agent writes the artifact file.
8. `openspec instructions apply --change add-dark-mode --json` returns concrete context files, task progress, task list, and dynamic apply state. The agent implements tasks and marks checkboxes.
9. `/opsx:verify` is implemented as generated skill instructions: the agent reads artifacts, searches the codebase, and produces a completeness/correctness/coherence report. There is no dedicated verification engine.
10. Archive has a real CLI path in `src/core/archive.ts`: it validates change delta specs, checks tasks, builds spec updates via `src/core/specs-apply.ts`, validates rebuilt specs, writes main specs, and moves the change into dated archive.

## Architecture

The system is a layered TypeScript CLI:

- CLI surface: `src/cli/index.ts` registers human and agent-compatible commands: `init`, `update`, `list`, `show`, `validate`, `archive`, `status`, `instructions`, `templates`, `schemas`, `new change`, `schema *`, and `workspace *`.
- Workflow schema engine: `src/core/artifact-graph/schema.ts`, `graph.ts`, `resolver.ts`, `state.ts`, `outputs.ts`, and `instruction-loader.ts` load schemas, validate DAGs, compute build order, detect completion from filesystem outputs, and produce enriched instructions.
- Built-in schema: `schemas/spec-driven/schema.yaml` defines proposal, specs, design, tasks, and `apply.requires/tracks`.
- Prompt and skill generation: `src/core/templates/workflows/*.ts` contains generated skill/command prompt text. `src/core/shared/skill-generation.ts` filters them by profile and renders `SKILL.md`.
- Tool adapter registry: `src/core/command-generation/registry.ts` registers adapters for Claude, Cursor, Codex, Gemini, GitHub Copilot, Windsurf, OpenCode, and many others. Adapters format files but do not deeply translate behavior to each agent runtime.
- Config/context loading: `src/core/project-config.ts`, `src/core/config-schema.ts`, and `src/utils/change-metadata.ts` parse global config, project config, schema selection, project context, per-artifact rules, and per-change schema metadata.
- Validation and merge: `src/core/validation/validator.ts`, `src/core/parsers/*`, and `src/core/specs-apply.ts` parse markdown specs/deltas, detect structural issues, and apply ADDED/MODIFIED/REMOVED/RENAMED requirements.
- Workspace coordination: `src/core/workspace/*` and `src/commands/workspace/*` manage multi-repo planning workspaces with shared `workspace.yaml`, local `local.yaml`, registry state, managed `AGENTS.md` guidance, and generated VS Code workspace files.
- Tests: `test/core/*` and `test/commands/*` cover artifact graph behavior, instruction loading, schema/config parsing, command generation, validation, archive behavior, profile drift, and workspace operations.

## Design Choices

- File-backed state instead of hidden databases. A change is complete when expected artifact files exist and tasks are checked in markdown.
- Schema-driven artifact graph. Schemas define artifact IDs, generated paths, templates, instructions, dependencies, and apply tracking.
- CLI as state oracle, agent as writer. The CLI produces structured context and guards; the agent reads files, writes artifacts, implements code, and reports progress.
- Context split into global background and artifact rules. Project config has `context` for all artifacts and keys such as `rules.proposal` for scoped constraints.
- Resolution order is explicit: schemas resolve project-local, then user override, then package built-in. Change schema resolves explicit flag, then `.openspec.yaml`, then project config, then `spec-driven`.
- Profile and delivery are separate. `core` installs only common workflows; `custom` can install more. Delivery controls whether generated artifacts are skills, commands, or both.
- Tool adapters are shallow. They adapt filesystem paths and frontmatter, not workflow semantics.
- Archive is more deterministic than sync. Archive uses code to apply delta specs and pre-validate rebuilt specs; `/opsx:sync` is described as agent-driven intelligent merging.
- Cross-platform filesystem handling is first-class. Code uses `path.join`, path canonicalization, Windows-aware move fallback, and tests for path handling.
- Workspace planning is intentionally cautious. Workspace guidance says linked repos are for exploration/planning and implementation edits require explicit user request.

## Strengths

- Strong reusable pattern: expose agent instructions as structured CLI JSON rather than static markdown only.
- Artifact graph gives agents a clear next action and dependency context without forcing a rigid phase model.
- Durable project memory is human-readable: specs, changes, tasks, and archives are all markdown/YAML.
- Per-artifact rules are a good context-control primitive; they avoid injecting spec-only guidance into design or task prompts.
- Validation catches many agent failure modes: missing delta sections, wrong scenario heading level, duplicate/cross-section conflicts, invalid schema DAGs, path traversal, and malformed config.
- Archive prepares and validates all spec changes before writing, which prevents partial corrupt updates.
- Tool support is broad and centralized; adding a new command surface usually means adding an adapter, not rewriting workflows.
- Tests cover important instruction-system invariants, including template parity hashes, profile drift, schema resolution, and workflow status JSON.

## Weaknesses

- Verification is mostly a prompt recipe. `/opsx:verify` asks the agent to search and infer coverage, but it does not have a structured evidence model, test runner contract, or reproducible scoring engine.
- File-existence completion can overstate progress. An empty or low-quality artifact can mark a graph node complete unless separate validation is run.
- Context injection is not token-optimal. A single `context` blob appears for all artifacts up to 50KB; there is no section-level retrieval, freshness, ranking, or compression.
- Tool adapters are syntactic. Generated prompts mention capabilities such as `AskUserQuestion`, `TodoWrite`, and `Task` that may not map cleanly to every supported assistant.
- Sync behavior can diverge from archive behavior. Generated `/opsx:sync` instructions emphasize agentic partial merging, while archive uses programmatic replacement/append rules.
- Some commands still rely on interactive prompts, making full automation uneven compared with JSON-friendly status/instructions/validate.
- Codex command files are installed globally under `$CODEX_HOME/prompts`, which can surprise teams expecting all generated workflow artifacts to be repo-local and versioned.
- Multi-agent support is mostly planning and prompt orchestration, not actual concurrency control or worker supervision.

## Ideas To Steal

- Build a small `instructions --json` command that returns exactly what an agent needs: output path, template, dependency files, project context, artifact rules, and unlocks.
- Treat workflow schemas as editable DAGs with `apply.requires` and `apply.tracks`, not hardcoded prompt phases.
- Store agent workflow state in simple files that both humans and agents can inspect.
- Separate `context` from `rules` and keep both out of generated artifacts by explicit instruction.
- Add a profile/delivery model so users can install a small default skill set and expand only when needed.
- Validate generated spec deltas before archive and validate rebuilt source specs before writing.
- Use archive history as durable memory and audit log, not just a cleanup step.
- Maintain workspace shared state and machine-local state separately to support cross-repo planning without committing absolute paths.
- Hash or snapshot generated prompt templates in tests to catch accidental prompt drift.

## Do Not Copy

- Do not rely on prompt-only verification for high-stakes implementation checks. Add machine-readable evidence, test command capture, and explicit pass/fail criteria.
- Do not mark artifact completion solely by file existence if artifact quality matters. Add content validation per artifact.
- Do not inject a large global context blob into every artifact without retrieval or budget controls.
- Do not assume all coding agents have equivalent question, todo, and subagent tools. Adapter layers should translate behavioral assumptions, not just frontmatter.
- Do not install workflow commands globally by default without a clear drift and versioning story.
- Do not keep two merge semantics for the same user concept (`sync` vs `archive`) unless the difference is explicit and tested.
- Do not let generated prompts become the only source of truth for critical logic. Keep deterministic operations in code when possible.

## Fit For Agentic Coding Lab

High fit as a source of reusable skills-instructions architecture.

Best direct applications:

- Schema-driven skills where each skill gets current instructions from a CLI or local resolver.
- Agent-facing JSON contracts for next actions, dependency context, and output targets.
- A lightweight artifact graph for planning/implementation/review workflows.
- Project config with concise global context plus per-artifact rules.
- Archive-time validation that turns agent-created artifacts into durable project memory.

Adoption should be selective. OpenSpec is optimized for a user-facing SDD product with broad tool integration. Agentic Coding Lab should steal the schema/instruction-loader/validation patterns, but add stronger verification evidence, tighter context retrieval, and clearer separation between deterministic CLI behavior and agent heuristics.

- `assets/*`: binary/image/SVG marketing and dashboard assets. Not relevant to reusable agent workflow logic.
- `src/ui/*`: welcome screen and ASCII UI. UI-only setup polish, not core instruction, context, verification, or orchestration behavior.
- `README_OLD.md`, `WORKSPACE_REIMPLEMENTATION_START_HERE.md`, `WORKSPACE_REIMPLEMENTATION_DIRECTION.md`, `openspec-parallel-merge-plan.md`: historical or planning documents superseded by current README/docs/code/specs. Skipped to avoid treating stale plans as implemented behavior.

- `flake.nix`, `scripts/update-flake.sh`, `build.js`, `scripts/pack-version-check.mjs`, `scripts/postinstall.js`, `scripts/test-postinstall.sh`: packaging/build/release mechanics. Skimmed only where installation/package behavior mattered; not central to agentic coding instruction patterns.

- `test/fixtures/**`: test data fixtures. Representative tests were read instead, which reveal fixture intent with less noise.
