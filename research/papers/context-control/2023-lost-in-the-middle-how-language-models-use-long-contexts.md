## Problem

The paper asks whether language models that accept long inputs actually use information throughout those inputs. This matters for retrieval-augmented QA, long-document processing, conversation history, and any agent workflow that grows a large prompt over time. A model can advertise a 16K, 32K, or 100K token window while still failing to use evidence that appears in an inconvenient position.

For Agentic Coding Lab, the same failure mode appears when a coding agent accumulates tool outputs, file listings, search results, plans, user constraints, and partial fixes. A crucial fact such as "do not edit this file", "the failing test is X", or "this branch has unrelated user changes" can become functionally unavailable if it is merely present but buried in the middle of a long context.

## Method

The paper uses controlled position interventions rather than only measuring performance at a fixed prompt layout.

The first task is multi-document question answering on NaturalQuestions-Open. For 2,655 questions whose annotated long answer is a paragraph, the authors build prompts with exactly one answer-containing passage and multiple distractor passages. The answer passage comes from the NaturalQuestions annotation; distractors are top Contriever MS-MARCO retrieval results that do not contain the annotated answer. They vary both the number of documents, with 10, 20, and 30 total documents, and the position of the answer passage.

The second task is synthetic key-value retrieval. The input is a JSON-like list of UUID key-value pairs plus a target key, and the model must output the matching value. The authors vary list length, with 75, 140, and 300 key-value pairs, and the position of the target pair. This strips away most semantic reasoning and isolates retrieval from input context.

The evaluated model set covers open and closed models available at the time: MPT-30B-Instruct, LongChat-13B (16K), GPT-3.5-Turbo, GPT-3.5-Turbo (16K), Claude-1.3, and Claude-1.3 (100K). Additional analyses compare encoder-decoder models, query-aware contextualization, instruction-tuned versus base models, Llama-2 sizes, and a GPT-4 subset.

The official code repository supports the paper's protocol. It provides `qa_data`, `kv_retrieval_data`, scripts for placing the gold answer at a chosen document index, scripts for generating UUID key-value data, prompt construction in `src/lost_in_the_middle/prompting.py`, experiment commands in `EXPERIMENTS.md`, and tests for prompt formatting.

## Evidence

The central result is a U-shaped position curve. Across many settings, models perform best when the relevant information is near the beginning or end of the context and worse when it is in the middle. In multi-document QA, GPT-3.5-Turbo drops by more than 20 percentage points in the worst cases, and its middle-position performance in 20- and 30-document settings can fall below its closed-book accuracy of 56.1%.

Longer context variants are not automatically better context users. When the same prompt fits in both a base and extended model variant, GPT-3.5-Turbo and GPT-3.5-Turbo (16K) have nearly overlapping position-sensitivity curves. Claude-1.3 and Claude-1.3 (100K) show the same kind of conclusion in the evaluated settings.

The key-value task confirms that this is not only a semantic QA issue. Claude-1.3 variants are near perfect, but GPT-3.5-Turbo, GPT-3.5-Turbo (16K), MPT-30B-Instruct, and LongChat-13B (16K) show failures or position sensitivity even when the task is exact retrieval over UUID pairs.

The ablations are important for practical design:

- Encoder-decoder models are comparatively robust when evaluated within their training-time sequence length, but they also show a U-shaped curve beyond that range.
- Query-aware contextualization, placing the query before and after the data, makes the synthetic key-value task nearly perfect for all evaluated models, including GPT-3.5-Turbo (16K) on 300 pairs. It does not meaningfully fix multi-document QA.
- Instruction tuning is not the whole cause. MPT-30B and MPT-30B-Instruct both show U-shaped curves, though instruction tuning improves absolute accuracy.
- GPT-4 has higher absolute QA accuracy on the sampled evaluation, but still degrades when relevant information is in the middle.
- Random distractors, randomized distractor order, and an unambiguous-question subset preserve the same qualitative trend.

The open-domain QA case study shows the cost side of the problem. Retriever recall continues improving as more documents are retrieved, but reader accuracy saturates much earlier. Moving from 20 to 50 retrieved documents gives only small gains, around 1.5 percentage points for GPT-3.5-Turbo and around 1 point for Claude-1.3, while increasing tokens, latency, and cost.

## Limits

The experiments are from the 2023 model generation. The result is field-defining and still heavily cited, but modern long-context models should be re-tested rather than assumed to have identical curves.

The tasks are QA and retrieval, not software engineering. Coding sessions involve multiple dependent facts, changing repository state, hidden user constraints, partial diffs, test artifacts, and tool failures. Those are harder than "one gold document" and not directly measured here.

The synthetic key-value task is intentionally narrow. It is useful for testing exact retrieval, but UUID lookup does not capture semantic compression, code navigation, multi-hop reasoning, or determining which fact is relevant.

The paper diagnoses and evaluates position sensitivity more than it provides a complete mitigation. It suggests reranking, truncation, and query-aware formatting, but it does not implement a full context manager, memory system, or agent policy.

The main QA metric checks whether an annotated answer appears in the output. That is appropriate for NaturalQuestions-style evaluation but does not measure citation quality, calibrated uncertainty, reasoning trace fidelity, or whether a generated answer used the supplied evidence.

## Research Themes

- Token efficiency: High relevance. The paper shows that adding more context can have sharply diminishing returns, so a smaller ranked context may outperform a larger indiscriminate one.
- Context control: Very high relevance. It is a core evaluation pattern for context assembly: same facts, different positions, measured behavioral change.
- Sub-agent / multi-agent: Low to medium relevance. It does not study multi-agent systems, but its findings support bounded handoffs and salient placement of delegated facts.
- Domain-specific workflow: Medium relevance. The experiments are general QA/retrieval, but the context-placement lesson transfers directly to coding workflows that need structured prompts, summaries, and handoffs.
- Error prevention: High relevance. Position-sensitivity tests catch failures that ordinary "the fact is somewhere in the prompt" reasoning misses.
- Self-learning / memory: Low relevance. It is not a memory paper, though it motivates active memory retrieval, reranking, and compaction policies.
- Popular skills: Medium relevance. The result can become a reusable skill rule: do not bury critical constraints and verification evidence in the middle of long prompts.

## Key Ideas

- Evaluate usable context, not nominal context length. A model's maximum window says little about how reliably it can use information at every position.
- Move the same relevant information to different positions and measure best-case versus worst-case performance.
- Treat the middle of long context as a risk zone for important evidence.
- Extended context windows need position-robustness tests. Bigger windows can preserve more tokens while still failing to use them.
- RAG quality depends on ordering and truncation, not just retriever recall.
- Query-aware contextualization can fix simple exact retrieval but is not a universal remedy for richer QA.
- Long-context evaluation should report positional robustness, not only average accuracy at one prompt layout.

## Ideas To Steal

- Add a context-position regression harness for Agentic Coding Lab. Take the same coding task and place critical facts at the start, middle, and end of a long transcript: user constraints, failing test output, target file path, prior failed hypothesis, and ownership boundaries. Score whether the agent still follows them.
- Make active context assembly position-aware. Put immutable user instructions, current goal, file ownership, and verification requirements near the front; put the latest state and next action near the end; avoid leaving critical one-off facts only in the middle.
- Use ranked truncation before raw expansion. For search results, code snippets, and command output, prefer fewer high-value chunks with provenance over dumping every plausible chunk into the prompt.
- Duplicate only the highest-risk facts. Repeating everything wastes context, but repeating destructive-operation constraints, current failing test names, and "do not touch" files can be cheaper than a wrong edit.
- Add "usable-context" metrics to memory and compaction evaluations: best-position score, worst-position score, and spread between them.
- Make subagent handoffs short and front-loaded. Each handoff should put acceptance criteria, file scope, exact commands, and known hazards in salient slots rather than relying on a long inherited transcript.
- For long tool outputs, restate the task-specific query before the output and extract an end summary of exact retained facts. The paper's query-aware result suggests this can help retrieval-like operations, even if it will not solve all reasoning tasks.

## Do Not Copy

- Do not assume the 2023 U-shaped curve magnitude applies unchanged to every current model. Use the paper as a testing template, not as a fixed constant.
- Do not replace all long-context use with aggressive truncation. Some coding tasks require distributed evidence across files, tests, and logs.
- Do not use UUID key-value retrieval as the only proxy for agent context quality. It misses semantic relevance, code dependencies, and changing state.
- Do not rely on query duplication as a blanket mitigation. It fixed synthetic retrieval in the paper but did not substantially improve multi-document QA.
- Do not conclude that relevant evidence at the end is always best. Recency helps in many settings, but stale or misleading recent content can still dominate the model.
- Do not optimize for retriever recall alone. More retrieved chunks can increase cost and distract the reader if context assembly is not controlled.

## Fit For Agentic Coding Lab

This paper is in-scope for context-control because it gives a simple, durable way to test whether a context strategy is working: hold information constant, vary placement, and measure behavior. For Agentic Coding Lab, the practical artifact is not a new memory implementation but a regression suite and prompt-assembly policy.

The main adoption path should be:

1. Define critical fact classes for coding agents: user constraints, file ownership, current objective, touched files, exact failing outputs, command history, hypotheses already rejected, pending risks, and next action.
2. Build long-context fixtures where each critical fact class is placed at the front, middle, or end.
3. Run candidate agent workflows against those fixtures and track worst-position degradation.
4. Update compaction, handoff, and retrieval policies so critical facts are placed in salient slots and irrelevant chunks are truncated before they bury state.

The key verdict for the lab is that "present in context" is not a sufficient safety or reliability guarantee. Context artifacts need placement, ranking, and regression tests.
